# Next.js Dashboard Plan

## Goal

Add a browser-based dashboard in a monorepo structure while keeping the current Python MVP as the source of truth for ingestion, topic extraction, scoring, fallback behavior, and persistence.

The first web version should:

- live inside this repo under `web/`
- use Next.js with TypeScript and the App Router
- read a JSON contract generated by Python
- support a manual refresh button
- be shaped so the JSON contract can later become a real HTTP API without rewriting the UI

## Architecture Direction

### Monorepo layout

```text
trend-friend/
  app/                    # Python ingestion, scoring, persistence
  scripts/                # Python entrypoints
  web/                    # Next.js app
  docs/
```

### Responsibility split

- Python remains responsible for:
  - external source fetching
  - fallback sample behavior
  - topic extraction
  - score calculation
  - SQLite persistence
  - exporting frontend-facing trend payloads
- Next.js becomes responsible for:
  - browser rendering
  - user interactions
  - filter state
  - refresh UX
  - future auth integration

This keeps business logic out of the frontend and makes the API migration straightforward.

## Data Contract Strategy

### Rule

Do not make the frontend depend directly on SQLite tables or Python dataclass shapes.

Instead, add a dedicated frontend-facing contract now and treat it as the future public API schema.

### Initial transport

Python will generate a JSON snapshot file for the UI, for example:

```text
web/data/latest-trends.json
web/data/trend-history.json
```

### Later transport

The same schema will be served over HTTP by a Python API, for example:

- `GET /api/trends/latest`
- `GET /api/trends/history`
- `POST /api/trends/refresh`

The frontend should consume these through a small data-access layer so switching from file reads to HTTP fetches is low-risk.

## Recommended Contract Shape

The schema should be cleaner than the current internal score model and should reflect product-facing language.

Example shape:

```json
{
  "generatedAt": "2026-03-09T21:08:16Z",
  "trends": [
    {
      "id": "ai-agents",
      "name": "AI Agents",
      "rank": 1,
      "score": {
        "total": 42.4,
        "social": 18.2,
        "developer": 16.1,
        "knowledge": 6.4,
        "search": 0.0,
        "diversity": 1.7
      },
      "sources": ["reddit", "github", "wikipedia"],
      "evidence": [
        "AI agents are replacing repetitive office workflows"
      ],
      "latestSignalAt": "2026-03-09T21:08:16Z"
    }
  ]
}
```

For history:

```json
{
  "generatedAt": "2026-03-09T21:08:16Z",
  "snapshots": [
    {
      "capturedAt": "2026-03-09T21:08:16Z",
      "trends": []
    }
  ]
}
```

## Refresh Flow

### Phase 1

- The dashboard exposes a refresh button.
- The button calls a Next.js server route.
- That route shells out to the local Python process during development.
- Python recomputes and rewrites the JSON snapshot.
- The route returns success/failure and the UI reloads the latest data.

### Later

- Replace the route implementation with a backend API call.
- Keep the client-side UI contract unchanged.

## Filtering Scope For V1

The first browser dashboard should support filtering by:

- source
- keyword
- minimum score

Keep filtering client-side initially because the payload size is small.
If history grows substantially, move filtering server-side later without changing the page structure.

## UI Scope For V1

Single dashboard page only.

Include:

- top trends list
- score breakdown columns
- source badges
- evidence preview
- last updated timestamp
- filter controls
- refresh button

Exclude for now:

- charts
- topic detail pages
- evidence drill-down views
- auth

## Styling Direction

Use:

- Next.js
- TypeScript
- App Router
- MUI Base

Recommendation:

- keep the component layer mostly custom
- use MUI Base primitives where they help with accessibility and state handling
- avoid coupling the product styling to a large opinionated component system early

## Persistence And History

History should be planned now, not bolted on later.

Implementation direction:

- keep the current SQLite storage as the operational source of truth
- add a historical snapshot export shape for the frontend
- ensure each pipeline run has a durable timestamp
- make the export layer able to read:
  - latest ranked trends
  - historical ranked snapshots

If the current persistence layer only stores the latest ranking, extend it to store timestamped runs before building history UI.

## API Readiness Requirements

To keep the JSON-first implementation easy to promote to a real API later:

- create a dedicated serializer module in Python
- avoid frontend knowledge of Python-only names or DB structure
- keep one canonical contract for both file export and future HTTP responses
- place refresh behavior behind a narrow interface in Next.js
- use stable IDs for topics
- version the contract if it starts changing materially

## Suggested Implementation Phases

### Phase 1: Contract and export layer

- define frontend DTOs in Python
- serialize latest trends to JSON
- serialize historical snapshots to JSON
- add a script to refresh exports

### Phase 2: Next.js scaffold

- create `web/` app
- set up TypeScript and App Router
- add base layout and dashboard page
- add a data access layer for local JSON reads

### Phase 3: Dashboard UX

- trends table/cards
- filter controls
- refresh button
- loading/error states

### Phase 4: History support

- add historical snapshot rendering
- support recent-run browsing
- prepare chart-ready structures even if charts wait for the next phase

### Phase 5: API migration path

- add Python API endpoints
- swap Next.js data access from file reads to HTTP
- keep page-level components unchanged

## Open Engineering Decisions

These do not block the plan, but they should be resolved during implementation:

- exact JSON file location under `web/`
- whether refresh in dev should invoke `main.py` or a narrower export script
- whether the single-container production shape keeps Next.js and Python in one process tree or uses a lightweight supervisor
- whether history is stored as complete ranked snapshots or as normalized per-topic time-series plus derived views
